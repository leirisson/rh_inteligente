Observações importantes para o design do frontend
ScreeningAnswer, ApplicationStage e Conversation não têm endpoints REST próprios na superfície atual — são manipulados internamente pelo application.service.ts (via processCandidateMessage). Se o frontend precisar exibir histórico de conversa ou respostas de triagem individualmente, esse endpoint ainda não existe — vale confirmar com o time de backend se está planejado.
Não existe endpoint de listagem de candidatos para a empresa (ex: GET /candidates global) — a única forma de a empresa "ver" candidatos é via GET /jobs/:jobId/matches (com threshold de similaridade) ou indiretamente por applications. Não há endpoint GET /applications (lista) nem GET /applications/:id (detalhe) — apenas POST /applications/:id/messages, o funil (/jobs/:jobId/funnel) e as rotas de interview. Isso é uma lacuna relevante para a tela de "pipeline de candidatos" — provavelmente o frontend precisará desse endpoint ainda não implementado, ou usar o funnel + matches como proxy.
Não há endpoint GET /users (lista de usuários do tenant) nem POST /users (convidar novo usuário) — só /users/me. Se o frontend precisa de gestão de equipe, esse endpoint também está ausente.
Todas as respostas de erro seguem o padrão { error: { message: string, code: string } } (visto em rbac.ts e webhook), com status 401 (não autenticado) / 403 (RBAC/tenant mismatch).
DELETE endpoints retornam 204 sem corpo (job, job-requirement, screening-question, contact-method).
Datas em request bodies usam z.coerce.date() (aceitam string ISO); em responses são serializadas como Date (verificar serialização real do Fastify — normalmente vira string ISO no JSON).
Paginação só existe em GET /jobs (page, pageSize, total) — outras listagens (requirements, screening-questions, matches, contact-methods, candidate applications) retornam arrays completos sem paginação.
